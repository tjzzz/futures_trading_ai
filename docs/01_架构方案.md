## 一、采集层 (Collector Layer)

### 1.1 采集器清单

| 采集器 | 类 | 频率 | 数据源 | 产出 |
|:------:|:--:|:----:|--------|------|
| `gold_silver.py` | `GoldSilverCollector` | 每5分钟 | gold-api.com | 快照 + minutely CSV |
| `yahoo_finance.py` | `YahooFinanceCollector` | 每5分钟 | Yahoo Finance API | 快照 + minutely CSV |
| `daily.py` | `TreasuryCollector` / `FREDCollector` / `VIXCollector` | 每天7:30 | Treasury/FRED/CBOE | 快照 + daily CSV |
| `rss_news.py` | 函数式 | 每30分钟 | CNBC/MarketWatch RSS | events JSON |
| `backfill_2026.py` | 函数式 | 一次性 | Treasury/FRED/CBOE | 回填 daily CSVs |
| `cleanup.py` | 函数式 | 每天凌晨 | — | 清理 minutely, 聚合 daily |

### 1.2 统一基类

所有采集器继承 `BaseCollector` ([collectors/base_collector.py](../collectors/base_collector.py))：

```
fetch() → parse() → run()
                     ├── write_snapshot() → dashboard_data.json
                     └── write_history_csv() → data/history/{grain}/xxx.csv
```

锁机制：写快照时用 `/tmp/` 下的排他文件锁（避免 iCloud O_EXCL bug）。高频采集器（yahoo_finance）直接批量读写绕过锁。

### 1.3 一键执行

[run_all_collectors.sh](../run_all_collectors.sh) 统一调度：

```bash
./run_all_collectors.sh              # 全部执行
./run_all_collectors.sh --historical # 仅回填历史
./run_all_collectors.sh --realtime   # 仅实时行情
```

## 三、分析层 (Analysis Layer)

### 3.1 统一入口

```python
analysis/engine.py::Analysis
    ├── analyze()              → 宏观页（四象限综合）
    ├── query_indicator()      → 归因页（回写作归因模块）
    ├── get_trend()            → 趋势数据
    └── handle_command()       → 飞书命令路由
```

根据 `ANALYSIS_MODE` 配置分发到 rules 或 llm 模式：

| 模式 | 配置 | 宏观页 | 归因页 |
|:----:|:----:|:------:|:------:|
| `rules` | 默认 | rules/macro.py 阈值规则 | attribution/ 统计归因 |
| `llm` | 需 LLM_API_KEY | llm/macro.py LLM 分析 | attribution/ + LLM 报告 |

### 3.2 宏观分析 (Macro Analysis)

**rules 模式** — [rules/macro.py](../analysis/rules/macro.py)：
- 四象限逐项分析：货币锚(TIPS/DXY)、流动性(利差)、风险偏好(VIX+新闻)、供需(金银比)
- 阈值驱动（10Y > 4.5% → bearish，VIX > 25 → bullish...）
- 加权信号综合 → 多空得分 → 场景标签（避险/利率压力/美元强势...）

**LLM 模式** — [llm/macro.py](../analysis/llm/macro.py)：
- 将 snapshot + 新闻组装成结构化 prompt
- 调用 OpenAI 兼容 API，要求返回 JSON 格式的四象限分析
- 输出格式与 rules 模式完全兼容，前端/飞书无缝切换

### 3.3 归因模块 (Attribution)

详见 [05_归因模块.md](05_归因模块.md)：

| 层 | 职责 | 方法 |
|:--:|:----:|:----:|
| L1 | 统计归因 | pearson r + 滚动 beta × Δ因子 |
| L2 | 事件匹配 | 异常波动日 ↔ S/A 级事件 |
| L3 | 报告生成 | 规则模板 或 LLM 叙事综合 |

L1 可独立上线，无需等待 L2/L3。

### 3.4 事件监控 (Event Monitor)

[event_monitor/monitor.py](../event_monitor/monitor.py)：

- 阈值定义：[thresholds.py](../event_monitor/thresholds.py) 中声明式配置（10Y > 4.5 告警、> 5.0 危机...）
- 24h 防重复推送（同级不重复，升级可覆盖）
- S 级 → 日志告警（预留飞书 @所有人），A 级 → 日志通知

### 3.5 预测模块 (预留)

架构预留位置，后续可扩展：

| 功能 | 说明 | 优先级 |
|:----:|------|:------:|
| 技术面信号 | MACD/RSI/布林带/趋势线 | 中 |
| 仓位管理 | 基于波动率的仓位建议 | 低 |
| 风险计算 | VaR / 最大回撤 / 压力测试 | 低 |

## 六、数据流全景

### 6.1 实时数据流

```
gold-api.com ──┐
               ├── gold_silver.py (5min) ──┐
Yahoo Finance ─┘                            │
                                            ├──→ dashboard_data.json ──→ Dashboard
Treasury CSV ──┐                            │
               ├── daily.py (7:30) ────────┘
FRED/CBOE ────┘
                                            ┌──→ data/history/minutely/ (7天)
                                            └──→ data/history/daily/ (永久)
```

### 6.2 归因查询流

```
用户选择品种+时间窗口
    ↓
attribution/engine.py
    ├── L1: statistical.py → 读 daily/minutely CSV → 因子贡献度排名
    ├── L2: event_matcher.py → 读 event_tracker → 匹配异动时刻
    ├── L3: report.py → 规则模板/LLM → 综合文本
    └── 返回结构化 JSON
```

### 6.3 事件推送流

```
cron / 手动触发
    ↓
event_monitor/monitor.py → check() → 读 dashboard_data.json → 对比阈值
    ↓
触发？→ 24h防重复检查 → S级/ A级 → 日志(→ 飞书预留)
```

### 6.4 飞书命令流

```
用户 @机器人 "宏观" / "归因 DXY 2026-04-01 2026-05-19"
    ↓
start.py → server.py → handler
    ↓
analysis/engine.py → handle_command()
    ├── "宏观" → analyze() → 格式化四象限报告
    ├── "归因" → query_indicator() → 格式化归因报告
    ├── "趋势" → get_trend() → 格式化趋势分析
    ├── "事件" → load_events() → 格式化事件列表
    └── "监控" → EventMonitor.get_active_thresholds()
    ↓
返回 markdown 消息 → 飞书群聊
```
