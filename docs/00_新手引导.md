> 适用：新加入的 AI Agent / 开发者
> 目标：快速了解项目全局，知道怎么接入开发，避免破坏现有架构

---

## 一、项目一句话

期货 AI 交易体系 — 以黄金（XAU）为核心的宏观基本面分析系统。采集多源数据 → 写入文件 → 分析引擎计算 → 展示在 Dashboard 或推送到飞书。

**核心哲学**：全通过文件解耦，没有消息总线，没有 Agent 通信框架。

## 三、开发铁律

### 规则 1：改自己的模块，别动别人的

```
collectors/    ← 采集器团队
analysis/     ← 分析引擎团队
dashboard/    ← 前端团队
feishu/       ← 飞书集成团队
event_monitor/ ← 事件监控团队
```

- **你负责 A 模块，只改 A 模块的文件**
- 需要改 B 模块才能完成你的功能？→ **停下来，把需求发出来让用户确认**
- 示例：你在做归因模块，发现 dashboard 的 attribution.html 需要改造 → 不要直接改，提交方案等用户确认

### 规则 2：保持全局架构一致

- **数据流方向不变**：采集器 → data/ 文件 → 分析引擎 → 展示层，永远是单向的
- **不改数据中台结构**：`dashboard_data.json` 的字段名、层级结构不能随意改，必须兼容现有消费方
- **不引入新的通信机制**：不要加消息队列、WebSocket（除非任务明确要求）、Agent 框架
- **新采集器必须继承 `BaseCollector`**，走 `fetch() → parse()` 模式

### 规则 3：页面样式一致性

- 颜色只用 `style.css` 的 CSS 变量，不硬编码色值（`--bg`, `--text`, `--blue`, `--teal` 等）
- 新增页面必须使用现有组件类：`.config-section`、`.page-header`、`.data-mgmt-table`、`.empty-state`
- 新增类名加前缀避免冲突：`.xxx-card`、`.xxx-table`
- 暗色主题整体风格不变：背景 `#0f1117`，卡片 `#161b27`，文字 `#e8eaf0`
- 涨跌颜色统一：涨 `--teal`（`#2dd4a0`），跌 `--coral`（`#ff6b6b`）

### 规则 4：API 命名规范

```
GET    /api/资源          → 列表/详情
POST   /api/资源          → 创建/查询
GET    /api/资源/子资源   → 子资源列表
```

- 不要用动词：`/api/getRealtimeData` ❌ → `/api/realtime/trend` ✅
- 不要用驼峰：`/api/realtimeData` ❌ → `/api/realtime/data` ✅

### 规则 5：SOP 流程不能跳

每个 SOP 文档末尾有检查清单（checklist），**逐项确认后才能提交流**：

- 采集器开发 → [检查清单](03_采集器开发.md#五检查清单)
- Dashboard 开发 → [检查清单](04_Dashboard开发.md#九检查清单)

## 四、典型开发流程

```
① 收到任务
   ↓
② 读相关文档（架构方案 + 对应 SOP）
   ↓
③ 确定改动范围
   ├── 只涉及自己模块 → 直接开发
   └── 跨模块改动    → 先向用户提交方案，确认后再开发
   ↓
④ 按 SOP 执行（逐项过 checklist）
   ↓
⑤ 提交代码 / PR，注明改了什么、是否涉及跨模块
```

### 跨模块改动示例

你在开发归因模块 L1，需要前端归因页展示新的归因结果。你应该这样说：

> "归因模块需要改造 attribution.html 来展示 L1 统计归因结果，涉及 dashboard/pages/attribution.html 和 app.js。改动范围：替换现有的 mock 数据渲染逻辑为新的 API 响应格式。是否可以改动？"

等用户确认后再动手。

## 五、常见问题

### Q: 新采集器的数据要写到 dashboard_data.json，会冲突吗？

A: 高频采集器（5分钟级）直接批量读写文件，不用锁。低频采集器用 `base_collector.write_snapshot()` 的排他锁。详见 [03_采集器开发.md](03_采集器开发.md#write_snapshot-vs-直接写快照)。

### Q: 新增的字段会不会影响现有页面？

A: dashboard_data.json 新增字段不会影响现有页面，现有页面只读自己关心的字段。但**修改已有字段名或结构**会破坏现有页面，必须确认。

### Q: 我想用 WebSocket / Redis / 消息队列 可以吗？

A: **不可以**，除非任务明确要求。项目的核心设计原则就是"全通过文件解耦"。引入新通信机制需要重新评估整个架构。

### Q: 怎么看当前开发进展？

A: 看根目录的 [README.md](../README.md#开发进展) 的开发进展表格和 [TODO.md](../TODO.md)。

## 六、架构速查

### 目录 vs 职责

| 目录 | 职责 | 读什么数据 | 写什么数据 |
|:----|:-----|:----------|:----------|
| `collectors/` | 从外部拉数据 | 外部 API / CSV | `data/` 下文件 |
| `analysis/` | 分析计算 | `data/` 下文件 | 不写文件（纯函数） |
| `dashboard/` | Web 展示 | `data/` + analysis | 不写文件 |
| `feishu/` | 飞书消息 | analysis 引擎 | 飞书 API |
| `event_monitor/` | 阈值告警 | `data/current/` | 日志 / 飞书 |

### 关键文件路径

| 文件 | 用途 |
|:----|:-----|
| `data/current/dashboard_data.json` | 所有指标最新快照 |
| `data/history/minutely/*.csv` | 分钟级历史（保留7天） |
| `data/history/daily/*.csv` | 日级历史（永久） |
| `data/events/event_tracker.json` | S/A 级事件时间轴 |
| `data/sources/source_registry.json` | 所有数据源注册信息 |
| `config.py` | 全局配置 |

## 七、多Agent协作质量检查清单

> 适用：代码把控管 / PR 评审者
> 目标：确保多Agent协作的代码质量一致性，避免架构破坏

### 必须检查项（PR 合并前必须满足）

- [ ] **架构一致性**：是否遵守"全通过文件解耦"原则，未引入新通信机制（消息队列、WebSocket、Agent框架等）
- [ ] **采集器规范**：新增采集器必须继承 `BaseCollector`（特殊情况需在PR描述中说明原因）
- [ ] **API设计**：符合RESTful规范，不使用动词，路径使用小写+连字符（如 `/api/realtime/trend`）
- [ ] **错误处理**：有具体的异常类型和清晰的错误信息，避免笼统的 `Exception`
- [ ] **日志记录**：使用统一日志配置，关键操作（fetch/parse/write）有适当的日志记录
- [ ] **数据中台兼容**：修改 `dashboard_data.json` 字段结构时，必须评估对现有消费方的影响
- [ ] **模块边界**：只修改自己负责的模块文件，跨模块改动需在PR描述中明确说明

### 建议检查项（鼓励满足，非强制）

- [ ] **类型提示**：函数参数和返回值有完整的类型提示，避免使用过于宽泛的 `Any` 类型
- [ ] **文档更新**：公共API变更需同步更新相关文档（README.md 或对应模块文档）
- [ ] **测试覆盖**：新增功能有相应的测试用例，至少包含正向用例
- [ ] **配置集中**：硬编码值已提取到配置文件（`config.py` 或专用配置文件）
- [ ] **代码复用**：相似功能已提取为公共工具函数，避免重复实现
- [ ] **性能考虑**：高频操作（如CSV读取）有适当的数据量限制和缓存机制

### 典型PR评审流程

```
① 收到PR通知
   ↓
② 检查PR描述是否清晰：改动了什么、为什么改、是否涉及跨模块
   ↓
③ 运行"必须检查项"清单（逐项确认）
   ↓
④ 如有问题 → 评论要求修改
   ↓
⑤ 通过检查 → 批准合并
```

### 常见问题处理

#### Q: Agent A 需要改 Agent B 的模块才能完成功能，怎么办？

A: 在PR描述中明确说明：
- 为什么需要跨模块改动
- 改动范围（具体文件、函数）
- 对现有功能的影响评估
- 等待代码把控管确认后再开发

#### Q: 新采集器的数据格式与现有格式不一致，怎么办？

A: 必须提供**数据迁移方案**或**兼容性处理**：
- 方案1：新增字段，不影响现有消费方
- 方案2：格式转换层，在采集器内部转换
- 方案3：版本化数据格式（需评估复杂度）

#### Q: 发现现有代码有问题（bug、性能瓶颈），但不是当前任务范围，要不要改？

A: **先记录，后评估**：
1. 在PR描述中备注发现的问题
2. 评估修复的紧急程度和影响范围
3. 如非紧急，建议创建独立的Issue/任务
4. 如紧急且影响当前功能，可与代码把控管协商处理方案
